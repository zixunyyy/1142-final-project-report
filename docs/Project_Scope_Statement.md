# EmpathEye 專案範疇說明書 (Project Scope Statement)

## 1. 專案概述
**專案名稱**：EmpathEye: AI 社交共鳴與情緒分析系統
**專案目標**：
- 提供文字輸入功能以記錄心情。
- 提供情緒回饋與分類結果（正向/負向/中性）。
- 提升使用者的情緒覺察與自我理解。
- 開發準確率達 85% 以上的即時文字情緒識別引擎。
- 建立視覺化「情緒溫度計」與「回覆建議系統」。

## 2. 目標使用者
1. 大學生、高中生
2. 有情緒紀錄或自我反思需求者
3. 遠距工作團隊（減少溝通誤解）
4. 社交焦慮或亞斯伯格症族群（輔助解讀情緒）
5. 線上客服或心理諮詢專業人員

## 3. 需求範圍分析

### 3.1 功能性需求 (In Scope)
- **F-01 情緒分析功能**：根據關鍵字判斷情緒類型（正向/負向/中性）。使用者輸入文字後，系統快速分析情緒色彩。
- **F-02 情緒紀錄查詢**：查看歷史輸入與分析結果。提供查詢介面，讓使用者回溯過去的心情點滴。
- **F-03 情緒統計展示**：以圖表呈現情緒比例與趨勢。透過圓餅圖或折線圖等視覺化工具呈現情緒分佈。

### 3.2 非功能性需求 (Non-functional Requirements)
- **效能**：系統需於 3 秒內完成輸入與分析。
- **安全性**：使用者資料僅限本地或資料庫存取，不外洩。
- **易用性**：介面簡單直覺，使用者可快速操作。

### 3.3 範圍限制 (Out of Scope)
- **無即時聊天或社群互動功能**：本系統設計為個人輔助工具，不提供人與人之間的直接傳訊或公開留言板。
- **無使用者登入多帳號系統**：系統不具備複雜的註冊、帳號管理機制，聚焦於本機或單一情境下的獨立使用。

## 4. 技術實作一致性評估

綜觀上述專案範疇，各項需求在技術實作上**具備高度一致性與可行性**。詳細評估如下：

### 4.1 資料庫與架構一致性 (無多帳號系統 vs 安全性)
由於「**不包含使用者登入多帳號系統**」，同時又要求「**使用者資料僅限本地或資料庫存取，不外洩**」，這在技術架構上完美契合。
**技術對應**：開發團隊可以採用輕量級的 SQLite 作為本地資料庫，只需建立一個 `Records` 資料表來儲存情緒紀錄，無需處理 Session、JWT token 或複雜的關聯式權限管理。大幅降低了後端開發的複雜度，是極佳的 MVP 範圍限縮策略。

### 4.2 效能與演算法一致性 (3秒內完成 vs 關鍵字判斷)
**需求**：「系統需於 3 秒內完成輸入與分析」及「根據關鍵字判斷情緒類型」。
**技術對應**：若採用關鍵字字典比對（Keyword-based Approach），或是輕量級的本機情緒分析模型（如 `SnowNLP`），其運算開銷極低，完全能在 0.5 秒以內回傳結果，輕鬆達成「3秒內」的效能要求。同時也因為不依賴外部大型雲端 API，避免了網路延遲，確保了效能與安全性一致。

### 4.3 介面與功能一致性
**需求**：「介面簡單直覺」、「以圖表呈現情緒比例與趨勢」。
**技術對應**：純前端（HTML/CSS/JS）配合輕量級圖表庫（如 Chart.js），能以極低的效能開銷與學習成本實現視覺化的「情緒溫度計」與趨勢圖，且無須引入大型框架即可快速構建直覺的操作介面。

### 總結
本專案的範疇定義非常精確。捨棄了「多帳號登入」與「即時通訊」兩個耗時且易生 Bug 的功能，讓團隊能把核心開發時間與精力完全集中在「**文字情緒識別引擎的準確率**」以及「**視覺化呈現的易用性**」上。這不僅降低了專案實作的風險，也能確保如期達成高品質的交付成果。
