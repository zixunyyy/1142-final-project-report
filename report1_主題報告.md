## 一、專題概述（Planning）

### 1.1 動機與背景

> 說明為什麼選擇這個主題，這個問題在生活中或產業中真實存在嗎？

隨著數位通訊普及，純文字互動缺乏語氣與表情，容易導致解讀偏差與社交誤解（研究顯示比例超過 50%）。本計畫旨在填補此「情感斷層」，不僅協助社交障礙者，也具備企業管理與遠距醫療的市場潛力。

### 1.2 目標使用者

> 這個系統是為誰設計的？他們有什麼特性或需求？

1. 大學生、高中生
2. 有情緒紀錄或自我反思需求者
3. 遠距工作團隊（減少溝通誤解）
4. 社交焦慮或亞斯伯格症族群（輔助解讀情緒）
5. 線上客服或心理諮詢專業人員

### 1.3 專題目標

> 完成後，這個系統能解決什麼問題？請用條列方式列出 3–5 點。

- 目標一：提供文字輸入功能以記錄心情。
- 目標二：提供情緒回饋與分類結果。
- 目標三：提升使用者的情緒覺察與自我理解。
- 目標四：開發準確率達 85% 以上的即時文字情緒識別引擎。
- 目標五：建立視覺化「情緒溫度計」與「回覆建議系統」。

## 二、需求分析（Requirements Analysis）

### 2.1 功能性需求（Functional Requirements）

> 系統「必須能做到」的事情。請列出核心功能（至少 3 項）。

| 功能編號 | 功能名稱 | 功能描述 |
| -------- | -------- | -------- |
| F-01     | 情緒分析功能 | 根據關鍵字判斷情緒類型（正向/負向/中性）。 |
| F-02     | 情緒紀錄查詢 | 查看歷史輸入與分析結果。 |
| F-03     | 情緒統計展示 | 以圖表呈現情緒比例與趨勢。 |

### 2.2 非功能性需求（Non-functional Requirements）

> 系統的品質要求，例如效能、安全性、易用性等。

| 類別   | 需求描述 |
| ------ | -------- |
| 效能   | 系統需於 3 秒內完成輸入與分析。 |
| 安全性 | 使用者資料僅限本地或資料庫存取，不外洩。 |
| 易用性 | 介面簡單直覺，使用者可快速操作。 |

### 2.3 範圍限制（Out of Scope）

> 明確說明這次專題「不會」做的功能，避免範圍膨脹。

- 不包含即時聊天或社群互動功能。
- 不包含使用者登入多帳號系統。

## 三、技術規劃（Technology Planning）

### 3.1 技術選型（Technology Stack）

| 層面 | 技術選擇 | 專業規格說明 |
| --- | --- | --- |
| **前端 (Frontend)** | HTML5 / CSS3 / Vanilla JavaScript | 採用無框架 (No-Framework) 的純前端架構，符合輕量化開發與課堂規範。CSS 用於刻畫直覺易用的 RWD 介面，JavaScript 負責動態互動與圖表資料渲染。 |
| **後端 (Backend)** | Python 3 + Flask | 使用輕量級的 Flask 框架建置路由，負責處理前端的文字輸入請求、執行關鍵字情緒判斷邏輯，並回傳分析結果。 |
| **資料庫 (Database)** | SQLite | 採用輕量化的關聯式資料庫，無須額外架設伺服器。資料檔存於本地端，確保使用者資料僅限本地存取，不外洩，符合專案安全性需求。 |
| **資料視覺化 (Data Viz)** | Chart.js | 導入此開源 JavaScript 圖表庫，將情緒歷史紀錄轉化為圓餅圖、趨勢圖等「情緒溫度計」與統計比例之視覺化呈現。 |
